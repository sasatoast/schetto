# 関連 Issue
<!-- 例: closes #123 -->

# 概要
マージ後に実現すること・変化することを簡潔に書く
関連仕様や参考文献があればリンクを添付

# 修正内容
本 PR で行った具体的な変更を箇条書きで
- 追加したファイル・機能
- 修正したロジック
- 削除したコード・設定

# 影響範囲
- 変更が及ぶ画面・API・DB・設定など
- 既存機能への影響の有無（破壊的変更かどうか）

# レビュー観点
レビュアーに重点的に確認してほしい箇所
- 設計・命名・責務の分離
- エッジケース・エラーハンドリング

# 検討内容
- 本 PR で採用した判断とその理由
- スコープ外にしたものと理由
  - 例: 既存モジュールと統一した、想定データ量から現状の方式で足りると判断した

# 実装メモ（任意）
コードから意図が伝わりにくい箇所の補足
図解・擬似コード・インラインコメントでの説明を推奨

# 確認内容
マージ可否の判断材料
  - 実施した動作確認
  - 他部署・外部のレビュー結果があれば記載

# 質問・相談（任意）
迷っている点、代替案の意見が欲しい箇所

# 補足
- デプロイ手順の変更があれば記載
- 環境変数・設定の追加があれば記載
- その他、レビュー時に知っておくとよい情報

# WIP について
WIP を外すために満たすべき条件
